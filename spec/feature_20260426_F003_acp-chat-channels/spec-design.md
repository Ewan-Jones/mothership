# Feature: 20260426_F003 - acp-chat-channels

## 需求背景

当前 RCS 已有 Environment、Agent、Instance 和 ACP 中继能力，但缺少“外部聊天通道”这一层抽象。用户最终希望在控制面中管理聊天通道，并把通道账号与某个 Agent 绑定，使微信、飞书等外部用户可以直接与 Agent 对话，而不需要先手工创建 Environment 和 Instance。

但首个版本先不接入任何具体平台：微信、飞书都不实现登录、收发消息、运行时桥接，也不落任何平台专属逻辑。当前版本只建立一层可扩展的“通道接入抽象层”，把控制面入口、后端接口骨架、平台适配器接口和运行时注册机制先搭好，后续版本再按平台逐个实现。

## 目标

- 在控制面顶部 Tab 新增 `通道` 入口，提供独立的通道管理页面
- 新增一层通用的 Channel Integration 抽象，统一描述不同平台的接入能力
- 后端建立平台适配器注册机制，为后续微信、飞书接入预留标准接口
- 前端新增“新增通道”弹窗，但首版中微信、飞书都置灰为“暂不支持”
- 首版不实现微信、飞书任何一个平台的登录、消息收发、Agent 绑定、Environment 创建、Instance 创建

## 非目标

- 不实现微信扫码登录
- 不实现飞书 OAuth / 扫码登录
- 不实现任何平台的消息监听、回发或运行时桥接
- 不实现通道与 Agent 的绑定流程
- 不实现基于通道的 Environment / Instance 自动创建

## 方案设计

### 一、交互入口与页面结构

控制面最上层 Tab 新增 `通道`，与现有配置类页面并列。进入后展示一个“通道接入”页面，首版页面包含：

- 顶部操作区：`新增通道` 按钮
- 空态说明区：说明“当前版本仅完成接入层抽象，平台接入将在后续版本逐步开放”
- 平台能力列表：展示当前已注册的平台适配器及其可用状态

`新增通道` 弹窗首版仅用于展示平台占位，不允许真正创建：

- `微信（暂不支持）`
- `飞书（暂不支持）`

两个入口都置灰，不发起任何后端创建请求。

### 二、通用接入层抽象

首版核心不是某个平台，而是一套统一的接入层接口。后端需要抽象出：

- **平台标识**
  - `wechat`
  - `feishu`
  - 后续可扩展更多平台
- **平台元数据**
  - 展示名称
  - 当前状态：`disabled | enabled`
- **平台适配器接口**
  - 获取平台元数据
  - 获取平台状态

首版所有平台适配器都只返回元数据和“未启用”状态，不提供任何真实接入实现。

### 三、运行时注册机制

为避免后续每接一个平台都直接往路由和页面里塞特例逻辑，首版要先建立一个统一注册层：

- 控制面后端通过注册表管理所有 Channel Integration Provider
- 每个 Provider 暴露统一的 descriptor
- 路由层只依赖注册表，不依赖某个平台实现细节
- 前端页面通过 API 读取平台列表和状态，而不是硬编码平台能力

注册表首版至少返回两条平台定义：

- `wechat`
- `feishu`

二者状态都为 `disabled`。

### 四、数据模型策略

由于首版不创建真实通道记录，也不落平台登录态，因此本版本不需要实现完整的 `channel` 业务实体和登录凭证持久化。

首版只需要持久化或暴露“平台能力描述”这一层信息，具体方式可以有两种：

1. **纯代码注册**
   - 平台列表由后端常量和注册表提供
   - 不写入数据库
2. **轻量持久化**
   - 只持久化 Provider 级配置和启用状态
   - 不持久化账号级 Channel Record

当前版本优先采用 **纯代码注册**，避免过早引入尚未使用的账号级 `channel` 表。

### 五、接口设计

首版建议新增 `web/channels` 相关只读/占位接口：

- `GET /web/channels/providers`
  - 返回平台接入列表
  - 示例字段：
    - `type`
    - `label`
    - `status`
- `GET /web/channels`
  - 返回当前通道列表
  - 首版固定返回空数组
- `POST /web/channels`
  - 首版保留接口骨架，但对所有平台返回 `409/400`，消息为“当前平台暂未开放”

首版不提供以下接口：

- `GET /web/channels/:id/login`
- `POST /web/channels/:id/bind-agent`
- `POST /web/channels/:id/unbind-agent`

这些接口留待后续真正实现平台接入时再加入。

### 六、前端交互细节

前端页面需要满足以下最小交互：

- 顶部 Tab 出现 `通道`
- 页面能显示“当前暂无可用通道接入”的说明
- `新增通道` 弹窗中展示：
  - `微信（暂不支持）`
  - `飞书（暂不支持）`
- 两个平台入口都不可点击提交
- 页面可展示后端返回的平台注册状态

首版不需要：

- 二维码弹窗
- 登录状态轮询
- Agent 绑定下拉
- Environment / Instance 关联展示

## 实现要点

1. **先做平台抽象，再做平台实现**  
   当前阶段最重要的是固定接口和注册机制，避免后续接微信时写一套、接飞书时再写另一套不兼容的逻辑。

2. **注册表优先于特例判断**  
   路由和前端都应面向统一 provider descriptor，不要在页面里写死“微信可扫码”“飞书未实现”这类平台特例分支。

3. **账号级 channel 实体延后**  
   既然首版不做登录、不做绑定，就没有必要提前落 `channel` 表和复杂状态机，避免产生大量暂时无用的数据结构。

4. **API 先保证稳定形态**  
   即便首版所有平台都 disabled，也应把 `/web/channels/providers` 与 `/web/channels` 的响应格式固定下来，后续只扩能力，不改协议形状。

5. **前端展示真实来源于后端**  
   页面上的“平台可用状态”应由后端 provider 注册表返回，前端只负责渲染，避免前后端状态漂移。

## 验收标准

- [ ] 控制面顶部新增 `通道` Tab，进入后可看到独立页面
- [ ] 通道页显示“当前版本仅完成接入层抽象”或等价说明
- [ ] 点击 `新增通道` 后可看到 `微信（暂不支持）` 和 `飞书（暂不支持）`
- [ ] 微信、飞书入口均为置灰状态，不能发起创建
- [ ] 后端存在统一的 Channel Integration Provider 注册接口
- [ ] `GET /web/channels/providers` 能返回 `wechat`、`feishu` 两个平台，且状态均为 `disabled`
- [ ] `GET /web/channels` 首版返回空数组，不报错
- [ ] 首版不包含二维码登录、消息桥接、Agent 绑定、Environment 创建、Instance 创建实现
